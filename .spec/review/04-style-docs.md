# Style & Documentation Issues: worktree-feature+read-sh-outline vs dev

| ID | Severity | File | Lines | Issue |
|----|----------|------|-------|-------|
| 1 | LOW | [scripts/render_outline.py](../../scripts/render_outline.py) | 168-176 | `format_metadata_value`'s final fallback is `return str(value)`. For a Python `bool` or `None` value pulled from a datalog query result (e.g. a `:block/collapsed?` field), `str(True)`/`str(False)`/`str(None)` renders as the Python literals `True`/`False`/`None` in the outline text, rather than the lowercase `true`/`false`/`null` that both the source data (JSON from Logseq's API) and any reasonable reader would expect. This is a minor cosmetic inconsistency, but since the entire point of this renderer is to present the JSON data in an easy-to-read form, leaking Python's `repr`-ish casing conventions into otherwise JSON-flavored output is a rough edge. Suggested fix: special-case booleans and `None` before the generic `str(value)` fallback, e.g. `if value is True: return "true"`, `if value is False: return "false"`, `if value is None: return "null"`. |
| 2 | LOW | [scripts/render_outline.py](../../scripts/render_outline.py) | 1-13 | The module docstring describes the two `--mode` shapes and the general reference/embed resolution behavior, but doesn't mention `--ref-depth` or the "references chase pointer chains to their final text regardless of `--depth`, bounded only by `--ref-depth` as a cycle guard" behavior implemented in `resolve_ref_text` (lines 87-103) — a design point the user specifically called out as important and that's also documented in `SKILL.md`. Anyone reading only this script's docstring (e.g. while debugging `read.sh`'s output, as the docstring itself suggests as its use case: "documented here for anyone debugging the output format" per `SKILL.md`'s Reference section) would miss this behavior entirely and could mistakenly believe `--depth` bounds all reference/embed resolution uniformly. Suggested fix: add a sentence to the module docstring, e.g. "Plain `((uuid))` references chase chains of pointer blocks to their final text regardless of `--depth`, bounded only by `--ref-depth` (a cycle guard, default 5) — `--depth` only limits nested block *embeds*." |
| 3 | LOW | [scripts/render_outline.py](../../scripts/render_outline.py) | 98, 115 | Inconsistent wording/formatting for analogous "couldn't resolve this" cases: an unresolved plain reference renders as the bare parenthetical `(unresolved reference)` (line 98, no surrounding `[...]((uuid))` wrapper, so it doesn't look like the other reference/embed output at all), while an unresolved embed renders as `[unresolved embed]((embed {uuid}))` (line 115, bracket-wrapped in the same style as a successfully-resolved reference/embed). A reader scanning the outline for the `[text]((uuid))` / `[text]((embed uuid))` pattern to spot references and embeds will visually miss the `(unresolved reference)` case since it doesn't match that pattern, and won't have the uuid available either (it's dropped entirely from that message, unlike the embed case which keeps `{uuid}` visible). Suggested fix: make the unresolved-reference case consistent with the unresolved-embed case, e.g. `return f"[unresolved reference]({{uuid}}))"` so both cases keep the uuid visible and follow the same bracket convention. |
