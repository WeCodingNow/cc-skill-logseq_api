# Review TODO: worktree-feature+read-sh-outline vs dev

Tracks open items from the branch review in `.spec/review/`. Nothing here has been fixed yet — this is a review-only pass, no code changes were made to `render_outline.py` or `read.sh` as part of it.

## Open

- [x] **B1 (HIGH)** — Embed-beyond-depth-budget fallback text gets reprocessed as a fresh reference by `REF_RE.sub`, corrupting output with nested brackets when the embed target is itself a pointer block. See [01-bugs.md](./01-bugs.md#b1-high----embed-beyond-depth-budget-fallback-text-is-reprocessed-as-a-fresh-reference-corrupting-output). Fixed in `df67427`.
- [x] **B2 (MEDIUM)** — `resolve_ref_text`'s pointer-chain detection only inspects the first line of a block's content, silently discarding any genuine multi-line pointer-block content (e.g. a reference annotated with a free-text line). See [01-bugs.md](./01-bugs.md#b2-medium----pointer-chain-resolution-only-inspects-the-first-line-silently-discarding-any-other-real-content-in-the-pointer-block). Fixed in `df67427`: no longer falsely chains past annotated blocks, though the annotation text itself still isn't surfaced (only the block's first line is shown) — a smaller remaining limitation than the original bug.
- [x] **B3 (MEDIUM)** — `call_logseq`'s `subprocess.run` has no `timeout=`, so a stalled Logseq API call hangs the entire render indefinitely. See [01-bugs.md](./01-bugs.md#b3-medium----no-subprocess-timeout-a-stalled-logseq-api-call-hangs-the-entire-render-indefinitely). Fixed in the commit that adds `timeout=10` and catches `subprocess.TimeoutExpired`.
- [x] **R1 (MEDIUM)** — No committed test fixtures despite the original plan specifying a fixture-based test suite; repo has zero test/fixture files, so the exact bugs found in this review (B1, B2) have no regression net going forward. See [02-refactoring.md](./02-refactoring.md#r1-medium----no-committed-regression-tests-despite-the-original-plan-specifying-a-fixture-based-test-suite). Fixed: added `scripts/fixtures/` (9 fixtures covering plain refs, chained refs, cycles, the B2 annotation case, embeds within/beyond the depth budget including the B1 repro, and both query shapes), a stubbed `call.sh`, and `run_tests.sh` — all 9 pass.
- [x] **R2 (LOW, partial)** — `TREE_METHODS` in `read.sh` omits `getSelectedBlocks`/`getPageLinkedReferences`/`getCurrentBlock`, so those allow-listed block-shaped methods silently fall through to raw JSON instead of the outline renderer. See [02-refactoring.md](./02-refactoring.md#r2-low----tree_methods-omits-other-block-shaped-read-methods-which-silently-fall-through-to-raw-json). Added `getSelectedBlocks` and `getCurrentBlock` (same `BlockEntity`/`BlockEntity[]` shape `render_tree` already handles). Deliberately left `getPageLinkedReferences` **out**: its actual response shape is `Array<[PageEntity, BlockEntity[]]>` (page+blocks pairs), which `resolve_child` doesn't unwrap — adding it would silently mis-render as `(no results)` instead of today's raw-JSON fallback. It needs its own row-shaped renderer; left as a follow-up, not attempted here.
- [x] **R3 (LOW)** — Three near-identical bash "scan array for match" loops in `read.sh` could be a single reusable function. See [02-refactoring.md](./02-refactoring.md#r3-low----three-near-identical-bash-scan-array-for-match-loops-could-be-a-single-reusable-function). Fixed: factored into a shared `array_contains` helper.
- [x] **Style #1 (LOW)** — `format_metadata_value`'s fallback uses Python-style `str(value)`, rendering booleans/`None` as `True`/`False`/`None` instead of JSON-style lowercase. See [04-style-docs.md](./04-style-docs.md). Fixed: special-cased `True`/`False`/`None` before the generic fallback.
- [x] **Style #2 (LOW)** — `render_outline.py`'s module docstring doesn't mention `--ref-depth` or the reference-chain-resolution behavior. See [04-style-docs.md](./04-style-docs.md). Fixed: added a paragraph documenting it.
- [x] **Style #3 (LOW)** — Inconsistent wording between `(unresolved reference)` (bare parenthetical, no uuid) and `[unresolved embed]((embed uuid))` (bracket-wrapped, uuid visible) for analogous "couldn't resolve" cases. See [04-style-docs.md](./04-style-docs.md). Fixed in `df67427`: the caller already wrapped `resolve_ref_text`'s return value in `[...]((uuid))`, so the uuid was never actually dropped as the original finding claimed — the real (smaller) fix was just normalizing the inner wording to plain `unresolved reference`, matching `unresolved embed`.
